React: Virtual DOM

O Virtual DOM é uma abstração em JavaScript criada para resolver um problema fundamental: como aplicar mudanças de estado na interface sem forçar o browser a recalcular o layout a cada operação.

React Internals · Reconciliation · Performance
O que é o Virtual DOM

Pense no Virtual DOM como uma cópia leve do DOM real, mantida inteiramente em memória como objetos JavaScript. Cada elemento da interface corresponde a um nó nessa árvore virtual — com tipo, propriedades e filhos, mas sem nenhum custo de renderização.

Cada nó do DOM tem um objeto leve correspondente no VDOM

No react, quando o estado de um componente muda, uma nova árvore virtual é gerada. É feito um processo de reconciliação - vamos falar dele logo adiante - comparando essa nova árvore com a anterior, calculando o conjunto mínimo de operações necessárias para sincronizar o DOM real. 1

O ponto importante aqui é que o browser não é chamado até que o diff esteja calculado. Todas as alterações são calculadas primeiro no Virtual DOM. O react agrupa um conjunto de atualizações, aplica no VDOM e depois modifica no DOM real somente aquilo que foi alterado, minimizando reflows e repaints. Porque isso? porque alterações no DOM real são custosas — cada mudança pode disparar um reflow, onde o browser precisa recalcular o layout de toda a página. O Virtual DOM é uma camada de otimização que permite ao React atualizar a interface de forma eficiente, exigindo o mínimo possível do browser - e obviamente do dispositivo, principalmente se for um device móvel.


O que é um reflow?

É o processo em que o browser recalcula a posição e o tamanho de todos os elementos afetados para reconstruir a árvore de layout. O browser possui um pipeline de renderização com etapas bem definidas:

Sempre que algo muda geometricamente, o browser precisa reexecutar a etapa de Layout. Qualquer operação que afete geometria ou layout dispara esse processo 2:

  • Alterar width, height, margin, padding ou border
  • Adicionar ou remover elementos do DOM
  • Alterar conteúdo textual
  • Redimensionar a janela
  • Ler propriedades como offsetWidth ou getBoundingClientRect() após mudanças pendentes

Esse custo explica os travamentos onde você tenta interagir com um elemento e nada acontece — o browser está realizando cálculos pesados e bloqueando a main thread.

O Virtual DOM tem justamente o objetivo de minimizar esse impacto, armazenando alterações em estruturas JavaScript, calculando apenas o que mudou e aplicando as mudanças no DOM real de maneira otimizada e em lote. E isso é feito através do algoritmo de reconciliação e que a partir do React 16 vem com o motor Fiber


Reconciliação — o algoritmo de diff

O React aplica heurísticas para transformar um problema O(n³) — comparação exata de duas árvores — em O(n), percorrendo cada nó apenas uma vez. 1

React Fiber — o motor atual

O algoritmo original (Stack Reconciler) tinha algumas problemáticas relevantes. Não conseguia priorizar processamentos/interações relevantes, o que por vezes fazia com que a exibição de um tooltip fosse mais importante que uma interação do usuário na tela. Além disso, ele percorria a árvore de forma recursiva e síncrona. Em árvores grandes, isso bloqueava a main thread por tempo suficiente para causar frames perdidos. 3

Com React 16 (setembro de 2017), o Fiber reescreveu o reconciliador do zero. A ideia central: cada componente vira uma fiber — um objeto JavaScript que representa um virtual stack frame, podendo ser pausado, retomado ou abortado. 3

Nas palavras de Andrew Clark, autor do Fiber:

"Fiber is a reimplementation of the stack, specialized for React components. You can think of a single fiber as a virtual stack frame. The advantage of reimplementing the stack is that you can keep stack frames in memory and execute them however and whenever you want."

— Andrew Clark, React Fiber Architecture 3

Work-in-progress tree

Durante uma atualização, o React constrói uma nova árvore de fibers em paralelo — sem tocar na árvore atual (current tree). Só quando o trabalho está completo a nova árvore é "comitada" para o DOM. 4
