Virtual DOM e React: por dentro do algoritmo que mantém sua UI eficiente

O Virtual DOM não é uma API do browser — é uma abstração em JavaScript criada para resolver um problema fundamental: como aplicar mudanças de estado na interface sem forçar o browser a recalcular o layout a cada operação.

React Internals · Reconciliation · Performance
O que é o Virtual DOM

Pense no Virtual DOM como uma cópia leve do DOM real, mantida inteiramente em memória como objetos JavaScript. Cada elemento da interface corresponde a um nó nessa árvore virtual — com tipo, propriedades e filhos, mas sem nenhum custo de renderização.

No React, quando o estado de um componente muda, uma nova árvore virtual é gerada. O algoritmo de reconciliação compara essa nova árvore com a anterior e calcula o conjunto mínimo de operações necessárias para sincronizar o DOM real. 1

A chave está em o que não acontece: o browser não é chamado até que o diff esteja calculado. As operações são agrupadas e aplicadas em lote — minimizando reflows e repaints.

O que é um reflow

É o processo em que o browser recalcula a posição e o tamanho de todos os elementos afetados para reconstruir a árvore de layout. O browser possui um pipeline de renderização com etapas bem definidas:

Sempre que algo muda geometricamente, o browser precisa reexecutar a etapa de Layout. Qualquer operação que afete geometria ou layout dispara esse processo 2:

  • Alterar width, height, margin, padding ou border
  • Adicionar ou remover elementos do DOM
  • Alterar conteúdo textual
  • Redimensionar a janela
  • Ler propriedades como offsetWidth ou getBoundingClientRect() após mudanças pendentes

Esse custo explica os travamentos onde você tenta interagir com um elemento e nada acontece — o browser está realizando cálculos pesados e bloqueando a main thread.

Layout thrashing

O código abaixo implementa um dos piores cenários possíveis, conhecido como layout thrashing: ler (offsetWidth) e escrever (style) no DOM dentro de um loop. A cada iteração, o browser invalida o cálculo anterior e precisa recomputar o layout, multiplicando o custo por n.

// ❌ Layout thrashing — O(n) reflows const elementos = document.querySelectorAll('.item'); elementos.forEach((e) => { if (10 > e.offsetWidth) { // ← leitura: força reflow e.style.backgroundColor = 'red'; // ← escrita: invalida layout e.style.width = '9px'; } });

Repare que ao solicitar offsetWidth, o navegador é forçado a recalcular o layout da página antes de retornar o valor — percorrendo a árvore do DOM, calculando posições e tamanhos, e bloqueando a main thread até que o cálculo seja concluído.

A solução: separar leituras de escritas

Leia todas as propriedades primeiro — o browser calcula o layout uma única vez. Depois aplique todas as modificações.

// ✅ Leitura separada da escrita — 1 reflow const larguras = [...elementos].map(e => e.offsetWidth); elementos.forEach((e, i) => { if (10 > larguras[i]) { e.style.backgroundColor = 'red'; e.style.width = '9px'; } });

💡 Para buscas pontuais, prefira getElementById: navegadores mantêm índices internos para IDs, tornando a busca O(1) em vez de O(n). 2

Reconciliação — o algoritmo de diff

O Virtual DOM tem justamente o objetivo de minimizar esse impacto, armazenando alterações em estruturas JavaScript, calculando apenas o que mudou e aplicando as mudanças no DOM real de maneira otimizada e em lote.

O React aplica heurísticas para transformar um problema O(n³) — comparação exata de duas árvores — em O(n), percorrendo cada nó apenas uma vez. 1

React Fiber — o motor atual

O algoritmo original (Stack Reconciler) percorria a árvore de forma recursiva e síncrona. Em árvores grandes, isso bloqueava a main thread por tempo suficiente para causar frames perdidos. 3

Com React 16 (setembro de 2017), o Fiber reescreveu o reconciliador do zero. A ideia central: cada componente vira uma fiber — um objeto JavaScript que representa um virtual stack frame, podendo ser pausado, retomado ou abortado. 3

Nas palavras de Andrew Clark, autor do Fiber:

"Fiber is a reimplementation of the stack, specialized for React components. You can think of a single fiber as a virtual stack frame. The advantage of reimplementing the stack is that you can keep stack frames in memory and execute them however and whenever you want."

— Andrew Clark, React Fiber Architecture 3

Telas a 60fps exigem que cada frame seja processado em menos de 16.6ms. O modelo síncrono do Stack Reconciler não conseguia garantir isso em árvores complexas — o Fiber resolve ao tornar o trabalho interruptível.

Isso permite ao React priorizar trabalho:

Alta prioridade → Animações Input do usuário
Baixa prioridade → Fetch de dados startTransition Renders em background

Work-in-progress tree

Durante uma atualização, o React constrói uma nova árvore de fibras em paralelo — sem tocar na árvore atual (current tree). Só quando o trabalho está completo a nova árvore é "comitada" para o DOM. 4
